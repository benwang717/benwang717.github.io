关于Q_OBJECT

发表于2023-12-02|更新于2023-12-22
|阅读量:

本篇主要针对涉及到Q_OBJECT的相关问题.

先上结论, 通常情况下, 在qt开发中当我们需要写一个功能接口的时候, 要尽量考虑一个问题:

即: 所写的接口是否需要兼容在纯C++项目中, 即需要正常使用在不能兼容Qt相关特性的环境中. 如果需要满足这个条件, 在添加接口的时候在头部(.h)就不能声明Q_OBJECT, 反之则需要声明.

然后需要解释一下Q_OBJECT是个什么东西, 在QT项目里添加新的文件(.h/.cpp)的时候, 会有很多预选项, 其中一项就是是否勾选[Q_OBJECT], Q_OBJECT在QT自带的字典中是这么解释的, “The Q_OBJECT macro must appear in the private section of a class definition that declares its own signals and slots or that uses other services provided by Qt’s meta-object system.”( Q_OBJECT宏必须出现在类定义的私有部分中，该类定义声明其自己的信号和插槽，或者使用Qt的元对象系统提供的其他服务。) 这里提到了元对象, “moc 全称是 Meta-Object Compiler，也就是“元对象编译器”。

Qt 程序在交由标准编译器编译之前，先要使用 moc 分析 C++ 源文件。如果它发现在一个头文件中包含了宏 Q_OBJECT，则会生成另外一个 C++ 源文件。这个源文件中包含了 Q_OBJECT 宏的实现代码。这个新的文件名字将会是原文件名前面加上 moc_ 构成。这个新的文件同样将进入编译系统，最终被链接到二进制代码中去。因此我们可以知道，这个新的文件不是“替换”掉旧的文件，而是与原文件一起参与编译。另外，我们还可以看出一点，moc 的执行是在预处理器之前。因为预处理器执行之后，Q_OBJECT 宏就不存在了。
Qt 将源代码交给标准 C++ 编译器，如 gcc 之前，需要事先将这些扩展的语法去除掉。完成这一操作的就是 moc。

我简单概括一下就是, 在我们需要调用具有QT独有特性的功能的时候, 如信号槽, 直接用的话C++是不认识的, 在这种情况下, 需要通过MOC作为一个”翻译”, 让MOC去向C++解释你的意图, 而Q_OBJECT就是这个”标志位”, 在你声明了Q_OBJECT的时候, 就表示你需要通过MOC向C++解释, 即你需要调用QT的特性功能.
